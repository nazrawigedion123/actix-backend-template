# Backend Actix Template

A production-ready Rust web API template using **actix-web 4.9** with a clean 3-layer architecture (Storage → Service → Handler), async Diesel ORM, structured JSON logging, and dependency injection.

## Prerequisites

| Tool | Version | Purpose |
|---|---|---|
| Rust | 1.85+ (edition 2024) | Compiler |
| PostgreSQL | 14+ | Database |
| Diesel CLI | 2.2+ | Migrations |
| Docker | 24+ (optional) | Local Postgres |
| Make | 4+ (optional) | Command shortcuts |

### Install Diesel CLI

```bash
cargo install diesel_cli --no-default-features --features postgres
```

## Quick Start

```bash
# 1. Start Postgres (or use your own)
docker compose up -d

# 2. Configure environment
cp .env.example .env   # or edit .env directly

# 3. Run database migrations
make db-setup

# 4. Start the server
cargo run
```

Server starts at `http://127.0.0.1:8080`.

## Architecture: 3-Layer Dependency Injection

Dependencies flow in one direction only: **Handler → Service → Storage**.

```
┌─────────────────────────────────────────────────────────────┐
│                    Composition Root                          │
│                   (src/initiator.rs)                         │
│                                                             │
│  1. Storage Layer   2. Module Layer    3. Handler Layer     │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐    │
│  │ DieselUser   │──▶│ DefaultUser  │──▶│ UserHandler  │    │
│  │ Repository   │   │ Service      │   │              │    │
│  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘    │
│         │                  │                  │            │
│         │ implements       │ implements       │ mounts     │
│         ▼                  ▼                  ▼            │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐    │
│  │ UserRepo     │   │ UserService  │   │  Routes      │    │
│  │ (trait)      │   │ (trait)      │   │  /api/v1/    │    │
│  └──────────────┘   └──────────────┘   └──────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

Each layer depends on **traits** (not concrete types), making every layer mockable for testing.

## API Endpoints

| Method | Path | Description | Auth |
|---|---|---|---|
| `POST` | `/api/v1/users` | Create a new user | None |
| `GET` | `/api/v1/users/{id}` | Get user by UUID | `x-api-token` header |

### Example Requests

```bash
# Create a user
curl -X POST http://localhost:8080/api/v1/users \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "email": "alice@example.com"}'

# Response: 201
{"success": true, "data": {"id": "uuid-here", "username": "alice", ...}}

# Get a user
curl http://localhost:8080/api/v1/users/<uuid> \
  -H "x-api-token: any-value-works"

# Response: 200
{"success": true, "data": {"id": "uuid-here", ...}}

# Get user without auth token
curl http://localhost:8080/api/v1/users/<uuid>
# Response: 401 Unauthorized
```

## Database Migrations

This template uses **Diesel CLI** for schema management. Migration files live in `migrations/`.

### Migration Commands

```bash
make db-status        # Check migration status
make migration-up     # Apply pending migrations
make migration-down   # Rollback last migration
make migration-redo   # Rollback + re-apply (development)
make db-setup         # Create DB + run all migrations

# Create a new migration:
make migration-create name=add_age_to_users
```

### How Migrations Work

1. Run `make migration-create name=<name>` — creates `migrations/<timestamp>-<name>/` with `up.sql` and `down.sql`
2. Write your SQL in both files
3. Run `make migration-up` — applies it and regenerates `src/internal/storage/generated/schema.rs`
4. Diesel CLI reads `diesel.toml` to know where to output the schema

## Adding a New Feature

Here is the complete workflow for adding a new domain (e.g. "Posts"):

### 1. Database Migration

```bash
make migration-create name=create_posts
```

Edit `migrations/<timestamp>-create_posts/up.sql`:

```sql
CREATE TABLE posts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id),
    title VARCHAR(255) NOT NULL,
    body TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Write the `down.sql`:

```sql
DROP TABLE posts;
```

Run `make migration-up`.

### 2. Storage Layer (`src/internal/storage/`)

Create `src/internal/storage/post_storage/mod.rs`:

```rust
use crate::internal::storage::generated::schema::posts;
use diesel::{Insertable, Queryable, Selectable};
use diesel_async::pooled_connection::bb8::Pool;
use diesel_async::AsyncPgConnection;
use serde::{Deserialize, Serialize};
use uuid::Uuid;

pub type DbPool = Pool<AsyncPgConnection>;

#[derive(Debug, Clone, Queryable, Selectable, Serialize, Deserialize)]
#[diesel(table_name = posts)]
#[diesel(check_for_backend(diesel::pg::Pg))]
pub struct PostModel {
    pub id: Uuid,
    pub user_id: Uuid,
    pub title: String,
    pub body: String,
    pub created_at: chrono::DateTime<chrono::Utc>,
    pub updated_at: chrono::DateTime<chrono::Utc>,
}

#[derive(Debug, Clone, Insertable, Deserialize)]
#[diesel(table_name = posts)]
pub struct NewPostModel {
    pub user_id: Uuid,
    pub title: String,
    pub body: String,
}
```

Add the `PostRepository` trait to `src/internal/storage/mod.rs`:

```rust
pub mod post_storage;

#[async_trait::async_trait]
pub trait PostRepository: Send + Sync {
    async fn find_by_id(&self, id: Uuid) -> Result<Option<PostModel>, diesel::result::Error>;
    async fn create(&self, new_post: NewPostModel) -> Result<PostModel, diesel::result::Error>;
}
```

Implement `DieselPostRepository` in `post_storage/mod.rs` (follow the `DieselUserRepository` pattern).

### 3. Service Layer (`src/internal/module/`)

Add the `PostService` trait to `src/internal/module/mod.rs`:

```rust
pub mod post_service;

#[async_trait::async_trait]
pub trait PostService: Send + Sync {
    async fn get_post(&self, id: Uuid) -> Result<PostModel, AppError>;
    async fn create_post(&self, user_id: Uuid, title: String, body: String) -> Result<PostModel, AppError>;
}
```

Create `src/internal/module/post_service/mod.rs` with your business logic.

### 4. Handler Layer (`src/internal/handler/`)

Create `src/internal/handler/post_handler/handler.rs` with actix-web endpoints.

### 5. Wire It Up in `src/initiator.rs`

```rust
// In AppInitiator::init():
let post_repo = Arc::new(DieselPostRepository::new(pool));
let post_service = Arc::new(DefaultPostService::new(post_repo));

// Add to Storage, Modules, Handlers structs
```

### 6. Register Routes in `src/internal/routes/mod.rs`

```rust
.route("/api/v1/posts", web::post().to(PostHandler::create_post))
.route("/api/v1/posts/{id}", web::get().to(PostHandler::get_post))
```

## Configuration

All configuration comes from environment variables (loaded via `.env` file):

```env
SERVER_PORT=8080               # Server listen port (default: 8080)
DATABASE_URL=postgres://...    # Postgres connection string (required)
RUST_LOG=info,backend_actix_template=debug  # Log level filter
```

The `RUST_LOG` variable controls verbosity. The format is:
- `info` — global default log level
- `backend_actix_template=debug` — your app module at debug level
- Set to `debug` or `trace` for full ORM query logging

## Logging

Structured JSON logging via the `tracing` ecosystem. Logs include:
- Timestamp
- Log level
- Module path
- Request span context (trace ID, etc.)
- Structured fields (e.g. `username`, `target_id`)

Log output format (to stdout):

```json
{"timestamp":"...","level":"INFO","target":"backend_actix_template","fields":{"message":"Starting actix production server cluster on port 8080..."}}
```
